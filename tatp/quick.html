Quick Start Guide for New TATP Users

Introduction

The purpose of this Quick Start Guide is to provide guidance to executing a trial run of TATP on a single computer. While this is not a reasonable configuration for production testing, a trial run will get you acquainted with TATP and the way it functions.

You need two Database Management System (DBMS) products. One is needed to run the Test Input and Result Database (TIRDB) and the other to run the System Under Test (SUT) database (that is, the one being tested). You can also use one DBMS product to run both, but we recommend selecting the product you are going to use with TIRDB already at this step.

Running TATP without TIRDB

This is the simplest way to run TATP. After the run, the resulting Mean Qualified Throughput (MQTh) in transactions per second) and the response time results will be displayed on the console and printed in the TATP log ('bin/tatp.log'). All the other information, such as throughput sampling, histogram data, and the session and run information, will be lost.
  1. Set up your target database.
  2. Create an ODBC Data Source Name (DSN) for the target database. (Note for Linux users: you do not have to worry about that if you use the solidDB ODBC library directly with the binaries included in the package. Otherwise, if you need a DBMS agnostic build, you have to build TATP with unixODBC first, and then configure a unixODBC data source)
  3. The instructions below assume that you will run TATP in the bin/ directory. Alternatively, you can setup a working directory and place your all test configuration files there.
  4. Comment out the line specifying TIRDB in the 'bin/tatp.ini' file,. The line starts with "TIRDBConnect =". The comment mark is "//".
  5. Study the Database Description Files (DDF) in 'ddf/. There is one file for each example target database. You can use any one of those, or create a new one to reflect your target database. Make sure that the transaction mix file corresponds to the product you are using as a target database. If there is no transaction file for the product you are using, use 'bin/tr_mix_generic.sql'.
  6. Write the DSN for the target database in the DDF file you have selected for execution. If no driver manager is used, enter the database connect string.
  7. Look at the 'tdf/example.tdf' file. It specifies a short test, with 1 minute ramp-up time and 3 minutes sampling time, using 4 client processes. It will also populate the target database with 10000 subscribers. The whole run will take about 4-5 minutes.
  8. In the 'bin' directory, run
    tatp ../ddf/<file you have picked> ../tdf/example.tdf
    The console output will produce the resulting MQTH values for each test run specified in the TDF file(s). If there are problems, analyze 'tatp.log' for possible errors. Note: it is really necessary to change the working directory to 'bin', before running TATP.

Running TATP with TIRDB

To include TIRDB in your run, proceed as follows:
  1. Set up an empty database designated to be TIRDB.
  2. Uncomment the line starting with "TIRDBConnect =" in the 'bin/tatp.ini' file.
  3. Create an ODBC Data Source Name (DSN) for TIRDB. Use "TIRDB", as it is preset in the configuration file 'bin/tatp.ini'. If you choose a different DSN, update 'tatp.ini' accordingly.
  4. Start TIRDB and, using any SQL query tool, execute the schema script 'tirdb/tirdb.sql'.
  5. Preload TIRDB with the data on your SUT and the target databases. The script 'tirdb/initDataExample.sql' preloads the necessary data on an example SUT platform and some database products. You can modify the script to add a product, or a version you would like to use in the trial.
  6. Execute the 'tirdb/initDataExample.sql' script. Your TIRDB is ready to run.
  7. Run TATP as described in the previous section.
  8. Retrieve the test results from TIRDB by using, for example, the statement:

    -- TATP Benchmark TIRDB query
    -- retrieve all TATP test runs
    SELECT
    se.session_id, session_name, test_run_id, test_name,
    te.start_date, te.start_time, te.stop_time,
    db_name, db_version, config_name, mqth_avg, subscribers
    FROM test_sessions se JOIN test_runs te
    ON se.session_id = te.session_id
    ORDER BY se.session_id DESC;

  9. Now, you can start thinking about organizing your production testing environment. The best location for TIRDB is a stationary, centrally accessed server, for best sharing of the test results. The client processes may be dispersed among various computers that must be specified in 'bin/remoteNodes.ini', in the main client computer. You do not need that file if you only use one computer to generate the load.
